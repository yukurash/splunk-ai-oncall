# シナリオ5 / Responder B（素の Claude・ツールなし）

入力＝アラート文のみ。subagent_tokens: 14992 / tool_uses: 0 / ~25s

## 根本原因（断定）
recommendation サービスの**メモリリーク（無制限に増え続ける内部キャッシュ）**。OTel Demo の recommendation に仕込まれた、product 一覧を際限なくキャッシュへ溜め込む不具合（`recommendationCache` 相当）が有効化されている状態。

## 根拠（推論）
- 「再起動直後は速い → 徐々に重くなる」は状態が時間とともに蓄積する典型。再起動でリセットされるのはプロセス内メモリ→外部依存障害ではない。
- 負荷スパイクや外部障害なら「ある時点から急に」だが、本件は経過時間に比例して単調悪化＝リーク/ヒープ肥大による GC 頻発・ページング増の挙動。
- 症状が recommendation に限局＝そのプロセス固有のリソース問題。
- OTel Demo の recommendation（Python）にキャッシュ無制限追加の故障シナリオがあるのは既知。

## 推奨対処
1. 即時緩和: recommendation をローリング再起動して latency を回復（応急）。
2. 根治: `recommendationCache` 系フラグを無効化。コード由来ならキャッシュに上限（LRU/TTL/maxsize）。
3. 検証: メモリ（RSS/ヒープ）の右肩上がり、再起動後の回復、OOMKill 履歴を確認。
4. 再発防止: キャッシュサイズ・メモリのアラート追加。

## 確信度（高）
「再起動でリセット＋経過時間に比例して悪化＋単一サービス限局」のメモリリーク三拍子が揃い、OTel Demo に該当シナリオが標準で存在するため。留保はツール不可で実メモリ推移を直接確認できない点。
