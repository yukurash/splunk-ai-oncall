# シナリオ5: recommendationCacheFailure（メモリリーク / 漸増劣化）

- flag: `recommendationCacheFailure` variant=on（recommendation の内部キャッシュが無制限肥大）
- 収集窓 -3m / Ground Truth: **recommendation のキャッシュ無制限増加によるメモリリーク（時間とともに遅くなる）**
- 証拠: **recommendation は APM 上は健全（p50=5ms / p99=19ms / err=0 / health=Ok）**。エラー/スロートレースにも未登場。数分のスナップショットでは劣化が映らない。

## 採点（初の分岐）
| 観点 | A: Claude+MCP | B: 素のClaude |
|---|:--:|:--:|
| 根本原因サービス | △ recommendation を最有力仮説に挙げたが**断定不可** | ✅ recommendation |
| 障害の性質 | △ メモリ漸増リークを仮説提示（データ未確認） | ✅ 無制限キャッシュのメモリリーク |
| ノイズ除外 | ✅ payment/flagd/llm の Critical を無関係と除外 | （対象外） |
| 確信度 | **低**（"今の窓では健全、劣化を確認できない"） | 高 |
| スコア(0-2) | **1** | **2** |

## 所見（記事の“限界/誤診”セクションの核）
- **メモリリークは APM のレイテンシ/エラーには即出ない**。固定の APM 証拠パックでは recommendation は健全に見え、A は正直に「断定不可」とした。B は症状（再起動で回復＋漸増）だけで“メモリリークあるある”を当てた。
- ただし **A の回答はより誠実で実務的**：早合点せず、Criticalノイズを除外し、「次に見るべきデータ＝メモリ時系列・コンテナOOM・内部スパン」を的確に列挙した。senior SRE の振る舞い。
- **重要**: Splunk MCP は**メモリメトリクスも保持**している（`o11y_get_metric_names` で `usedmemory*` 等が取得可）。今回 A は固定の APM パックしか渡されず気づけなかった。**完全自律でツールを選べる A なら、A 自身が言ったとおりメモリを取りに行って確証できた**はず。
- 教訓: 「データを渡す」だけでは不十分。**何のデータを取りに行くかを AI 自身に選ばせる（自律ツール選択）**ことが、レイテンシ/エラー以外の障害（メモリ・飽和）に効く。
